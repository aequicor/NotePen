# Requirements: Main Screen Redesign

**Module:** common
**Status:** Draft
**Date:** 07.05.2026
**Author:** @Main

---

## Business Context

В текущем состоянии приложение NotePen открывает PDF-редактор без стартового экрана,
что вынуждает пользователя каждый раз вручную выбирать файл через файловый менеджер.
Это особенно неудобно при регулярной работе с одними и теми же документами.

Переработка главного экрана решает две проблемы:
1. **Быстрый доступ к недавним файлам** — список последних открытых документов с
   визуальным превью первой страницы позволяет мгновенно идентифицировать нужный файл
   без навигации по файловой системе.
2. **Единая точка входа** — кнопка открытия нового файла через нативный проводник
   платформы доступна прямо со стартового экрана.

Экран должен одинаково хорошо работать на Android-смартфонах и на Desktop (Windows,
macOS, Linux) в рамках Compose Multiplatform, адаптируясь к размеру и форм-фактору
экрана.

## Acceptance Criteria

### История файлов — хранение и обновление

- [ ] AC-1. Given приложение запущено впервые, when пользователь открывает главный экран, then список недавних файлов пуст и отображается состояние «нет файлов» с подсказкой открыть первый документ.
- [ ] AC-2. Given пользователь открыл PDF-файл, when файл успешно открыт в редакторе, then путь к этому файлу сохраняется в локальной истории.
- [ ] AC-3. Given файл сохранён в истории, when пользователь открывает главный экран, then этот файл присутствует в списке недавних.
- [ ] AC-4. Given пользователь открывает файл, который уже есть в истории, when файл открыт, then запись перемещается в начало списка (дублирования не возникает).
- [ ] AC-5. Given в истории уже хранится максимальное количество записей (default: 20), when пользователь открывает ещё один файл, then самая старая запись удаляется, новая добавляется в начало. *(OQ-1 закрыт default=20; PO может переопределить значение — см. Open Questions)*
- [ ] AC-5a. Given файл открыт повторно через нативный файловый диалог (не из списка), when файл уже присутствует в истории, then применяется та же дедупликация, что и в AC-4: существующая запись перемещается в начало, новая запись не создаётся. Сравнение путей выполняется после нормализации: разрешение символических ссылок (canonical path), удаление trailing slash, регистронезависимо на платформах с case-insensitive файловой системой (Windows, macOS по умолчанию) и регистрозависимо на Linux.
- [ ] AC-5b. Given приложение работает на Android и файл выбран через Storage Access Framework (URI вида content://), when запись добавляется в историю или проверяется на дубликат, then первичным ключом дедупликации служит content:// URI as-is (без нормализации пути файловой системы). Дополнительно сохраняются displayName и размер файла (bytes) как вторичные атрибуты для fuzzy-match: если два URI отличаются (например, после реавторизации SAF выдал новый URI), но displayName совпадает и размер файла идентичен — приложение предлагает объединить записи (существующая перемещается в начало, новый URI обновляет первичный ключ). Окончательное решение об объединении принимает пользователь — автоматического тихого слияния не происходит.

### Отображение списка недавних файлов

- [ ] AC-6. Given список недавних файлов не пуст, when пользователь открывает главный экран, then список отображает записи в порядке убывания времени последнего открытия (самый свежий — сверху).
- [ ] AC-7. Given элемент списка отрисован, when он отображается на экране, then для него показана миниатюра первой страницы PDF-документа.
- [ ] AC-8. Given миниатюра ещё не сгенерирована, when элемент списка виден пользователю, then отображается плейсхолдер (заглушка) на время генерации.
- [ ] AC-9. Given файл из истории был удалён или перемещён на устройстве, when пользователь видит его в списке, then элемент визуально помечен как недоступный (файл не найден).
- [ ] AC-9a. Given главный экран открывается (или становится видимым после возврата из редактора), when приложение отображает список недавних файлов, then доступность каждого файла проверяется асинхронно в фоне — UI не блокируется; список сразу отображается с предыдущим статусом записей, и статус каждой записи обновляется по мере завершения проверки. Медленные сетевые диски не задерживают первичный рендер экрана. Фоновая периодическая проверка не требуется.
- [ ] AC-9b. Given файл в списке недоступен и помечен как таковой, when недоступных записей несколько, then они не вытесняют доступные записи из лимита AC-5 — недоступные и доступные записи совместно занимают общий лимит (20); порядок определяется временем последнего открытия.
- [ ] AC-9e. Given все 20 слотов истории заняты записями, помеченными как недоступные, when пользователь открывает новый рабочий файл (доступный PDF), then самая старая недоступная запись (по времени последнего открытия) вытесняется из списка, а новый файл добавляется в начало. Доступные записи никогда не вытесняются в пользу новых, пока в списке есть хотя бы одна недоступная запись.
- [ ] AC-9c. Given PDF-файл повреждён или зашифрован (открыть нельзя), when приложение пытается открыть его из списка, then элемент помечается состоянием «ошибка файла» (отличным от «файл не найден»), а пользователь получает соответствующее сообщение об ошибке; запись остаётся в списке и пользователь может повторить попытку открытия позднее (запись не удаляется автоматически). *(Default: retain-with-retry; PO может переопределить на auto-remove — см. OQ-3)*
- [ ] AC-9d. Given миниатюра файла не может быть сгенерирована из-за ошибки рендеринга (повреждён/зашифрован), when элемент списка виден пользователю, then отображается плейсхолдер ошибки, визуально отличающийся от плейсхолдера «в процессе генерации» (AC-8).
- [ ] AC-10. Given элемент списка доступен, when пользователь нажимает на него, then приложение открывает соответствующий PDF в редакторе.
- [ ] AC-11. Given элемент списка помечен как недоступный, when пользователь нажимает на него, then пользователь получает понятное сообщение о том, что файл не найден, и запись остаётся в списке.

### Открытие нового файла

- [ ] AC-12. Given главный экран открыт, when пользователь нажимает кнопку открытия нового файла, then открывается нативный файловый диалог платформы.
- [ ] AC-13. Given нативный диалог открыт, when пользователь выбирает PDF-файл и подтверждает выбор, then приложение открывает файл в редакторе и добавляет его в историю.
- [ ] AC-14. Given нативный диалог открыт, when пользователь отменяет выбор, then приложение остаётся на главном экране без изменений.

### Навигация и обновление списка

- [ ] AC-17. Given пользователь возвращается с экрана редактора на главный экран, when главный экран становится видимым, then список недавних файлов обновляется: файл, который редактировался, отражён в актуальной позиции с актуальным статусом доступности. На Desktop обновление списка происходит при двух событиях: (1) закрытии окна редактора и (2) при первом показе главного экрана (начальный рендер после запуска приложения). Фоновые проверки доступности между этими событиями не требуются.
- [ ] AC-18. Given запись об открытом файле добавляется в историю, when запись сохраняется, then операция сохранения выполняется до того, как пользователь получает возможность взаимодействовать с редактором (flush при открытии, а не при закрытии файла). Если flush завершился ошибкой, редактор всё равно открывается — ошибка сохранения истории не блокирует работу с документом; при этом пользователь получает ненавязчивое уведомление о том, что запись не попала в историю. *(Default: non-blocking flush; потеря истории не блокирует редактирование)*

### Многооконный режим Desktop

- [ ] AC-19. Given приложение на Desktop открыто в нескольких окнах одновременно, when пользователь открывает файл в одном из окон, then запись об этом файле добавляется в общую историю, доступную во всех окнах.
- [ ] AC-20. Given два файла открыты одновременно в разных окнах Desktop с минимальным временным интервалом, when оба события записываются в историю, then порядок определяется фактическим временем добавления записи; конфликт разрешается без потери данных (последний пишет — последний в топе). При неразличимых временных метках (разница < 1 мс) победитель определяется детерминированным лексикографическим сравнением идентификаторов сессий окон — порядок предсказуем и повторяем. *(Требует PO: должен ли идентификатор окна/сессии быть введён как понятие; если нет — допустим произвольный, но стабильный тай-брейк)*

### Адаптивная вёрстка

- [ ] AC-15. Given приложение запущено на Android-смартфоне (ширина экрана < 600 dp), when главный экран отображается, then список файлов представлен в одну колонку.
- [ ] AC-16. Given приложение запущено на Desktop или планшете (ширина экрана ≥ 600 dp), when главный экран отображается, then список файлов представлен в несколько колонок (сетка).

### Жизненный цикл и восстановление

- [ ] AC-21. Given ОС Android завершила процесс приложения (process death), when пользователь заново запускает приложение, then список недавних файлов отображает те же записи, что были до завершения процесса (история персистентна между запусками).
- [ ] AC-22. Given миниатюры файлов кешируются локально, when суммарный размер кеша миниатюр превышает 50 МБ, then наиболее давно использованные записи кеша вытесняются (LRU); точный лимит определяется в технической спецификации, но принцип вытеснения обязателен. Кеш миниатюры инвалидируется при изменении mtime (времени модификации) файла на диске: при следующем отображении элемента миниатюра перегенерируется, если mtime файла новее mtime кешированного изображения.

### Папки

- [ ] AC-24. Given главный экран открыт, when пользователь создаёт папку, then он вводит произвольное имя; папка появляется на главном экране под секцией недавних файлов.
- [ ] AC-25. Given папка создана, when пользователь открывает список недавних файлов, then папка не изменяет состав и порядок элементов в этом списке.
- [ ] AC-26. Given PDF-файл присутствует в истории, when пользователь добавляет его в папку, then файл отображается внутри этой папки; запись о файле в истории недавних файлов остаётся без изменений.
- [ ] AC-27. Given PDF-файл добавлен в папку, when пользователь добавляет тот же файл в другую папку, then файл присутствует в обеих папках одновременно; каждая папка хранит независимую связь с этим файлом.
- [ ] AC-28. Given PDF-файл находится в папке, when пользователь убирает его из этой папки, then файл перестаёт отображаться в папке; запись о файле в истории недавних файлов и его принадлежность к другим папкам не изменяются.
- [ ] AC-29. Given папка содержит PDF-файлы, when пользователь удаляет папку, then папка исчезает с главного экрана; все PDF-файлы, которые были в этой папке, остаются в истории недавних файлов и доступны как прежде.
- [ ] AC-30. Given PDF-файл удалён из всех папок, when пользователь просматривает историю недавних файлов, then файл по-прежнему присутствует в истории (принадлежность к папкам не влияет на историю).
- [ ] AC-31. Given на главном экране есть папки, when пользователь просматривает экран, then секция недавних файлов расположена выше секции папок.
- [ ] AC-32. Given диалог создания папки открыт, when поле имени пусто или содержит только пробельные символы, then кнопка подтверждения неактивна (заблокирована) и создать папку невозможно.
- [ ] AC-33. Given уже существует папка с некоторым именем, when пользователь создаёт новую папку с тем же именем, then создание разрешено; обе папки существуют одновременно и различаются внутренним идентификатором системы, а не именем.
- [ ] AC-34. Given диалог создания папки открыт, when пользователь вводит имя длиной более 255 символов, then ввод символов сверх лимита блокируется; поле не допускает сохранение имени длиннее 255 символов.
- [ ] AC-35. Given диалог создания папки открыт на Android и пользователь сворачивает приложение, when пользователь возвращается в приложение, then диалог отображается снова с тем текстом, который был введён до сворачивания.
- [ ] AC-36. Given диалог создания папки открыт на Desktop и пользователь переключается на другое окно или сворачивает приложение, when пользователь возвращается в приложение, then диалог отображается снова с тем текстом, который был введён до переключения.
- [ ] AC-37. Given пользователь пытается добавить файл в папку, when файл отсутствует в истории недавних файлов, then добавление невозможно; папки работают исключительно поверх истории недавних файлов и не предоставляют независимого механизма добавления произвольных файлов.
- [ ] AC-38. Given PDF-файл в истории помечен как недоступный (файл не найден на диске), when пользователь добавляет его в папку или файл уже находится в папке, then операция разрешена; внутри папки файл отображается с тем же статусом «недоступен», что и в истории недавних файлов.
- [ ] AC-39. Given пользователь инициирует удаление папки, when запрос на удаление получен, then приложение показывает диалог подтверждения; папка удаляется только после явного подтверждения пользователем; отмена диалога оставляет папку без изменений.
- [ ] AC-40. Given на главном экране нет ни одной папки, when пользователь просматривает главный экран, then секция папок не отображается; after удаления последней папки секция папок исчезает немедленно.
- [ ] AC-41. Given количество папок достигло 100, when пользователь пытается создать новую папку, then кнопка создания папки заблокирована и снабжена подсказкой о достижении лимита; создать папку сверх лимита невозможно.
- [ ] AC-42. Given история недавних файлов пуста, when на главном экране присутствует хотя бы одна папка, then секция недавних файлов отображается с пустым состоянием (согласно AC-1) выше секции папок; секция папок отображается ниже.
- [ ] AC-43. Given файл вытеснен из истории (список достиг лимита и появился более новый файл), when пользователь открывает папку, в которой этот файл находился, then файл остаётся в папке и доступен для открытия напрямую из папки; запись о файле в истории недавних файлов отсутствует, но связь папки с файлом сохраняется. *(Вариант b: «архивный» файл)*
- [ ] AC-44. Given файл присутствует в папке в статусе «архивный» (вытеснен из истории), when пользователь открывает этот файл из папки, then файл открывается в редакторе и добавляется обратно в историю как вновь открытый (с дедупликацией по AC-4/AC-5a/AC-5b). При upsert архивного файла в полную историю (20 слотов) применяется тот же порядок вытеснения, что и в AC-9e: сначала вытесняется самая старая недоступная запись; если недоступных нет — самая старая доступная (стандартный LRU).
- [ ] AC-45. Given папка содержит файл, when пользователь переименовывает папку, then папка отображается под новым именем; все файлы, находившиеся в папке, и связь между ними сохраняются без изменений.
- [ ] AC-46. Given диалог создания папки открыт, when пользователь нажимает Back (Android) или Escape (Desktop), then диалог закрывается без создания папки и без сохранения введённого текста.
- [ ] AC-47. Given на главном экране отображается список папок, when пользователь видит этот список, then папки упорядочены по дате последнего открытия или изменения любого PDF внутри каждой папки (самая «активная» папка — сверху); при открытии или редактировании файла из папки позиция этой папки обновляется.
- [ ] AC-48. Given диалог создания папки открыт, when пользователь вводит имя, then допустимы только Unicode-буквы (категория \p{L}, включая кириллицу и диакритику), Unicode-цифры (\p{N}), дефис («-») и нижнее подчёркивание («_»); недопустимые символы не вводятся и не отображаются в поле ввода.
- [ ] AC-49. Given пользователь вставляет текст (clipboard paste) в поле имени папки, when вставляемый текст является многострочным, then берётся только первая строка; затем из неё удаляются все символы, не соответствующие \p{L}+\p{N}+[-_], и результат подставляется в поле ввода (с соблюдением лимита 255 символов из AC-34). Если после фильтрации первой строки результат оказывается пустым, в поле подставляется пустая строка; кнопка OK остаётся заблокированной согласно AC-32; следующие строки вставляемого текста не анализируются и не подбираются.

- [ ] AC-50. Given файл в папке одновременно является архивным (вытеснен из истории) и недоступным на диске (файл не найден), when этот файл отображается внутри папки, then он получает отдельное визуальное состояние «архивный + недоступен», визуально отличающееся как от состояния «только архивный», так и от состояния «только недоступен»; конкретная иконка определяется @Designer.
- [ ] AC-51. Given диалог переименования папки открыт, when пользователь вводит новое имя, then применяются те же правила валидации, что и при создании: поле не может быть пустым или содержать только пробельные символы (AC-32), длина ограничена 255 символами (AC-34), допустимы только Unicode-буквы, цифры, дефис и нижнее подчёркивание (AC-48).
- [ ] AC-52. Given папка содержит исключительно архивные файлы (ни один из файлов не присутствует в истории недавних файлов), when система определяет позицию папки в списке (AC-47), then для сортировки используется `lastOpenedAt` каждого файла, сохранённый в связи FolderFileLink на момент вытеснения; папка не опускается в конец списка из-за отсутствия файлов в актуальной истории.
- [ ] AC-53. Given файл внутри папки имеет статус «архивный + недоступен» (AC-50), when пользователь нажимает на этот файл, then приложение показывает сообщение о недоступности файла (аналогично AC-11 для недоступных файлов в истории); архивный статус в модели сохраняется без изменений. Сообщение об «архивности» не показывается — в UI доминирует признак «недоступен».
- [ ] AC-54. Given файл присутствует в папке и вытесняется из истории недавних файлов (лимит AC-5 достигнут), when вытеснение происходит, then приложение не показывает пользователю никакого предупреждения; файл молча переходит в статус «архивный» внутри папки (AC-43). Секция недавних файлов обновляется согласно AC-5.
- [ ] AC-55. Given папка содержит файлы в статусе «архивный» (вытеснены из истории), when пользователь удаляет эту папку, then папка удаляется с главного экрана; все архивные записи, связанные исключительно с этой папкой (не присутствующие в истории и не состоящие ни в одной другой папке), остаются в модели данных, но нигде не отображаются на главном экране — ни в истории недавних файлов, ни в секции папок.
- [ ] AC-56. Given папка существует и в ней нет ни одного файла (все файлы убраны или папка только что создана), when пользователь просматривает главный экран, then пустая папка отображается в секции папок и показывает счётчик «0 файлов»; автоматического удаления пустой папки не происходит.

### Позиция прокрутки

- [ ] AC-57. Given пользователь открыл PDF-файл из истории или через нативный диалог и перешёл на определённую страницу в редакторе, when пользователь покидает редактор (закрывает документ или возвращается на главный экран), then номер последней просматриваемой страницы сохраняется вместе с записью файла в истории.
- [ ] AC-58. Given в записи истории сохранён номер последней страницы, when пользователь открывает этот файл из списка недавних файлов, then редактор открывает документ на сохранённой странице, а не на первой. Если сохранённый номер страницы превышает фактическое количество страниц документа (например, файл был изменён), редактор открывается на последней доступной странице.

### Временная метка

- [ ] AC-23. Given файл добавляется в историю, when момент записи определяется, then временем открытия считается момент, когда пользователь инициировал открытие файла (нажал на элемент списка или подтвердил выбор в диалоге), а не момент завершения загрузки в редакторе.

## Out of Scope

- Дизайн экрана (цвета, отступы, типографика, конкретные компоненты) — разрабатывается отдельно через @Designer.
- Функция удаления отдельной записи из истории пользователем.
- Функция полной очистки истории пользователем.
- Поиск или фильтрация в списке недавних файлов.
- Сортировка списка по критериям, отличным от времени последнего открытия.
- Синхронизация истории между устройствами или через облако.
- Поддержка форматов документов, кроме PDF.
- Предпросмотр страниц, кроме первой.
- Открытие файлов из системы «Недавние файлы» операционной системы (OS-level recent files) — только внутренняя история приложения.
- Механизм пинирования (закрепления) файлов в начале списка.
- Вложенные папки (папка внутри папки) — текущий скоуп поддерживает только один уровень; вложенность является будущим расширением.
- Drag-and-drop файлов на главный экран Desktop — вне текущего скоупа; открытие через нативный диалог является единственным входным путём для новых файлов.
- Автоматическое удаление записей о недоступных или повреждённых файлах из истории (auto-remove after N launches) — явно вне скоупа; текущее поведение: retain-with-retry (OQ-3, OQ-7).
- Call-to-action кнопка непосредственно в пустом состоянии списка — пустое состояние содержит только текстовую подсказку; единственная кнопка открытия файла доступна через основной элемент управления на экране (вопрос 7 — решение требует PO, см. Open Questions).

## Dependencies

- **Платформенный файловый диалог** — нативный проводник Android (Storage Access Framework) и Desktop; реализуется через expect/actual в Compose Multiplatform.
- **PDF-рендеринг (превью)** — зависит от существующего движка рендеринга PDF в модуле `common` (`:app:byCompose:common`); миниатюры генерируются на базе того же механизма.
- **Локальное хранилище истории** — новая подсистема, реализуется с нуля; механизм хранения (DataStore, SQLite, файл) определяется в технической спецификации.
- **@Designer** — UI/UX дизайн главного экрана разрабатывается параллельно и является входными данными для реализации вёрстки.
- **Модуль `:shared`** — доменные модели и бизнес-логика хранения истории могут размещаться в shared-модуле; граница определяется в технической спецификации.
