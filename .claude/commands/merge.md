# /merge

Закрывает worktree-сессию **NotePen**: коммитит ветку фичи, прогоняет тесты и
сливает её **merge-коммитом** в основную ветку. Локально, **без push**. Деструктивных шагов нет, но
основную ветку трогаем только после подтверждения.

> Для командных репозиториев с общей основной веткой безопаснее открыть PR, а не лить локально. Эта
> команда рассчитана на соло-/пет-проекты, где основную ветку ведёшь ты один.

## Шаги

1. **Сориентируйся.** Основная ветка = `$1`, иначе дефолтная ветка репозитория
   (`git symbolic-ref --short refs/remotes/origin/HEAD` без префикса `origin/`; если её нет — `main`
   или `master`, что реально существует). Текущая ветка — `git branch --show-current`. Если ты уже на
   основной ветке — **СТОП**: `/merge` льёт из ветки фичи В основную, не наоборот.

2. **Закоммить работу ветки.** Посмотри `git status` и `git diff`. Есть незакоммиченное — просмотри
   изменения и **убедись, что не тащишь секреты/ключи/мусор** (в этом бандле нет хука guard-commit,
   гейт на тебе), затем закоммить осмысленным сообщением. Нечего коммитить — иди дальше; пустой коммит
   не делай.

3. **Прогон тестов.** `./gradlew allTests` (долго — в фоне с опросом статуса). Красные — **СТОП**: покажи падения, основную ветку не трогай.

4. **Подтверждение.** Покажи сводку: какая ветка → в какую, список коммитов
   (`git log <основная>..HEAD --oneline`), и что будет merge-коммит **без push**. Дождись явного «да».

5. **Merge (`--no-ff`, всегда отдельный merge-коммит).** Основная ветка почти наверняка занята
   (checked out) в другом worktree, и git не даст слить в занятую ветку из этого каталога. Поэтому:
   - Найди её worktree: `git worktree list`.
   - Занята где-то — мёржи прямо в том каталоге, не переключая текущий: сначала проверь, что он чистый
     (`git -C <path> status`; грязный — **СТОП**, попроси пользователя разобраться), затем
     `git -C <path> merge --no-ff <ветка-фичи>`.
   - Нигде не занята — сделай checkout основной ветки в корневом worktree и мёржи там.
   - Используй `--no-ff` даже если возможен fast-forward.

6. **Конфликты → СТОП.** Не разрешай вслепую и не делай `git merge --abort` без спроса. Покажи
   конфликтные файлы и оставь состояние пользователю.

7. **Push — никогда.** Только локальный merge. Пуш и удаление worktree-ветки — отдельные явные
   действия пользователя (не вызывай их сам).

## Формат отчёта

```
## merge
<ветка-фичи> → <основная> | merge-коммит <sha> | tests: green
осталось: запушить вручную (git push), при желании убрать worktree
```

Остановился на тесте/конфликте/подтверждении — скажи прямо, на каком шаге и почему.
