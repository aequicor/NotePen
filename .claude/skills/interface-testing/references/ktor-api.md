# Тестирование HTTP API Ktor

Плейбук стадии «тестирование интерфейсов» для Ktor-сервера. UI нет — «интерфейс» это HTTP-эндпоинты.

## Предпочтительно: testApplication

Без поднятия реального порта, быстро и детерминированно (зависимость `ktor-server-test-host`):

```kotlin
@Test
fun `GET items returns 200 with body`() = testApplication {
    application { module() }                 // твой Application.module()
    val response = client.get("/items")
    assertEquals(HttpStatusCode.OK, response.status)
    assertTrue(response.bodyAsText().contains("\"items\""))
}
```

- Тестируй через сконфигурированный `module()`, а не «голый» сервер — иначе проверяешь не то, что в проде.
- Для JSON-клиента в тесте подключи `ContentNegotiation` к тест-клиенту так же, как на сервере.

## Что проверять

| Кейс | Ассерт |
|---|---|
| Позитивный | корректный статус (200/201), форма тела, заголовки (`Content-Type`) |
| Плохой ввод | 400/422, тело с понятной ошибкой, **нет** стектрейса наружу |
| Без авторизации | защищённый маршрут отдаёт 401/403, а не 200 |
| Не найдено | 404 на несуществующий id |
| Идемпотентность/конфликт | повтор POST → ожидаемое поведение (409/идемпотентный 200) |

## Живой сервер (если testApplication не применим)

1. Подними локально: `./gradlew run` (в фоне, опрашивай статус/лог).
2. Бей по эндпоинтам HTTP-клиентом, проверяй статус/тело/заголовки.
3. Обязательно прогони и негативные кейсы (4xx), не только happy-path.
4. Останови сервер после прогона.

## Дисциплина

- Проверяй **реальный** ответ (статус + тело), а не факт «запрос ушёл».
- Не коммить отладочные правки (открытый CORS, выключенный TLS, логирование тела) — это для локального прогона.
- Падение → возврат к стадии разработки с шагом воспроизведения (метод, путь, тело, фактический vs ожидаемый ответ).
