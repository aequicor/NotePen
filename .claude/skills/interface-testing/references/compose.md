# UI-тестирование через claude-in-mobile MCP

Плейбук стадии «тестирование интерфейсов» для Compose Multiplatform / Android / iOS / desktop. Требует подключённого MCP-сервера `mobile` (`npx -y claude-in-mobile`) и платформенных инструментов под цель: `adb` (Android), Xcode + simctl (iOS), JDK + Gradle (desktop).

> Точные имена инструментов бери из схемы MCP `mobile` в текущей сессии — ниже описаны возможности, а не сигнатуры.

## 1. Поднять приложение

| Цель | Как запустить |
|---|---|
| desktop-Compose | найди run-таску: `./gradlew tasks --all` → обычно `:desktopApp:run` / `:composeApp:run` / `runDistributable`. Запусти в фоне. |
| Android | собери `./gradlew assembleDebug`, поставь на эмулятор/устройство (`adb install`), запусти; либо через инструмент запуска MCP. |
| iOS | подними симулятор (simctl), установи и запусти; MCP сам поднимет WebDriverAgent для дерева доступности. |

Дождись, пока приложение реально откроется (снимок/дамп подтверждает), прежде чем действовать.

## 2. Прочитать экран

- Сделай **дамп дерева** (`ui-dump`/`analyze-screen`) — получишь структурированный список элементов (текст, тип, bounds, доступность).
- Находи целевые элементы **по тексту/семантике**, а не по координатам. Для нечёткого совпадения — `find-and-tap` с порогом уверенности.

## 3. Пройти сценарий

1. Действие: тап по элементу (по тексту/индексу из дампа) или ввод текста.
2. Снова дамп.
3. **Ассерт наблюдаемого результата**: появился ожидаемый текст/экран, изменилось состояние. Не «кнопка нажата», а «открылся экран X с текстом Y».
4. Повторяй до конца сценария.

## 4. Дисциплина

- **Блок-лист деструктивных действий**: не нажимай `delete` / `logout` / `reset` / оплату / необратимые операции без явной задачи. Сомневаешься — пропусти и отметь в отчёте.
- Никаких фиксированных `sleep` — жди по состоянию (повторный дамп до появления элемента).
- Не привязывайся к абсолютным координатам — экраны/плотности разные.

## 5. Базовая доступность

- Размер тач-таргетов: ≥ 48dp (Android) / 44pt (iOS).
- У интерактивных элементов есть label/contentDescription.
- Разумный порядок фокуса.

## Отчёт

Сценарий → результат (pass/fail) → доказательство (скриншот/дамп/наблюдаемый факт) → проблемы (где, как воспроизвести). Падение — это возврат к стадии разработки, а не «починю тест».
